## 1. 전체 사용자 흐름 (Flow)

이 흐름은 '사용자가 생각하지 않고 말하면 자동으로 기록되도록' 하는 것을 최우선 목표로 설계되었습니다.

**Happy Path (성공적인 기록 흐름):**

1.  **Initiation (시작):** 사용자가 하단 중앙의 🎤 버튼을 탭한다.
2.  **Recording (녹음):** 시스템이 음성 녹음을 시작하고, 즉시 "듣고 있어요" 상태를 표시한다.
3.  **STT & NLP (처리):** 음성이 텍스트로 변환(STT)되고, LLM/NLP 모듈이 텍스트를 분석하여 **의도(Intent)**, **구조화된 데이터(Entities)**, **확신 점수(Confidence Score)**를 추출한다.
4.  **Structure Generation (구조화):** 추출된 정보(예: 수유, 120ml, 시간)를 기반으로 최종 기록 초안을 생성한다.
5.  **Validation & Prompt (확인 요청):** 시스템은 분석 결과의 불확실성(Confidence Score)에 따라, 구체적인 확인 질문을 생성하여 사용자에게 제시한다. (예: "수유 120ml 기록할까요?")
6.  **User Confirmation (사용자 응답):** 사용자는 [확인] 또는 [수정] 버튼 중 하나를 선택한다.
7.  **Final Save (최종 저장):** 사용자의 확인에 따라 데이터가 최종적으로 DB에 구조화되어 저장된다.
8.  **Feedback (피드백):** 성공적인 기록 완료 메시지를 사용자에게 제공한다.

**Error/Ambiguity Path (예외 처리 흐름):**

1.  **STT Failure:** 음성 인식이 불가능할 경우 → "다시 말씀해 주세요" 메시지 출력 및 S1(대기)로 복귀.
2.  **Low Confidence:** NLP 모델이 의도를 확신하지 못할 경우 → **즉시 5단계(Validation & Prompt)**로 진입하여 확인 요청을 한다.
3.  **Missing Entity:** 필수 정보(예: 수유량)가 누락된 경우 → **보완 요청**을 통해 사용자에게 필요한 정보를 재요청한다.

## 2. 상태 정의 (State)

시스템이 실시간으로 관리해야 할 핵심 상태 목록입니다.

| 상태 ID | 상태 명칭 | 설명 | 주요 액션 |
| :--- | :--- | :--- | :--- |
| **S1** | **대기 (Idle)** | 앱이 기본 상태로 대기하며, 음성 입력 버튼을 기다리는 상태. | 🎤 탭 대기 |
| **S2** | **녹음 중 (Recording)** | 사용자가 음성 입력하는 동안 STT 및 실시간 분석을 진행하는 상태. | 음성 입력 감지 |
| **S3** | **분석 중 (Analyzing)** | STT 결과가 텍스트로 변환된 후, LLM/NLP가 의도, 엔티티, 점수를 분석하는 내부 처리 상태. | (시스템 내부 처리) |
| **S4** | **확인 요청 (Confirming)** | 시스템이 분석 결과를 바탕으로 사용자에게 구체적인 확인 질문을 제시하는 상태. | [확인] / [수정] 선택 대기 |
| **S5** | **저장 중 (Saving)** | 최종 확인이 완료된 데이터가 DB에 쓰여지는 외부 API/DB 통신 상태. | (시스템 내부 처리) |
| **S6** | **완료 (Success)** | 모든 기록이 성공적으로 저장되고 피드백이 완료된 최종 상태. | 화면 표시 후 S1로 복귀 |

## 3. 에러 케이스 (Error Cases)

시스템이 예상치 못한 입력이나 처리 실패에 대응하기 위한 구체적인 처리 방안입니다.

| 에러 유형 | 발생 조건 | 시스템 대응 전략 |
| :--- | :--- | :--- |
| **STT 실패** | 음성 입력이 시스템에 의해 인식되지 않음 (잡음, 낮은 볼륨 등). | 즉시 사용자에게 "다시 말씀해 주세요" 메시지를 출력하고 S1 상태로 복귀. |
| **의미 불명확** | LLM/NLP가 사용자의 의도를 확신하지 못함 (낮은 Confidence Score). | **S4 (확인 요청)** 상태로 즉시 전환하여 "무슨 기록인지 잘 모르겠어요"와 같은 모호한 메시지 대신 구체적인 질문을 제시. |
| **데이터 누락** | 필수 엔티티(예: 수유량, 시간)가 음성에서 추출되지 않음. | **보완 요청**을 통해 누락된 정보를 사용자에게 명확하게 요청하여 S4 상태로 전환. |
| **구조화 실패** | 추출된 엔티티를 DB 스키마에 매핑하지 못함. | 내부 로깅을 통해 실패 원인을 기록하고, 사용자에게 "기록에 문제가 발생했습니다. 잠시 후 다시 시도해 주세요" 알림. |
| **타임스탬프 오류** | 시간 정보의 추출 또는 기록에 오류가 발생함. | 시스템 내부에서 타임스탬프를 재검증하고, 오류 발생 시 사용자에게 재입력을 요청. |

## 4. UX 화면 정의 (UI Definition)

사용자 경험을 최우선으로 하여, 복잡한 입력 과정 없이 직관적인 확인을 유도합니다.

**A. 입력 중 화면 (Recording Phase)**

*   **상태 표시:** 화면 중앙에 녹음 진행 상황을 명확히 표시.
    *   **상태:** "듣고 있어요" (음성 입력 중)
    *   **가이드:** "기록할 내용을 말씀해 주세요."
*   **디자인 원칙:** 입력 중에는 모든 UI 요소를 최소화하고 음성 입력에 집중하도록 한다.
